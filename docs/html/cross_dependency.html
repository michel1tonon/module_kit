CrossDependency<RETURN, PARAM>

Simple contract for module-to-module communication without importing internals.

Why It Matters

  • Encourages self-contained modules.
  • Avoids direct feature coupling.
  • Eases evolution to multi-package architecture.

Architectural View

  • Treat CrossDependency as a boundary port between modules.
  • The contract belongs to a stable shared layer, not to feature internals.
  • The owner feature provides the adapter/implementation.
  • Consumer features depend only on the contract, preserving dependency direction.
  • This keeps modules replaceable, testable, and ready for micro frontend evolution.

Practical Contract + Implementation

abstract class SelectCountryCross
    implements CrossDependency<Future<String?>, SelectCountryInput> {}

class SelectCountryCrossImpl implements SelectCountryCross {
  @override
  Future<String?> call(SelectCountryInput param) async {
    // owner module flow
    return 'BR';
  }
}

Recommended Boundary

  • Keep the contract in shared.
  • Keep implementation in the owner feature module.
  • Consumers should depend on the contract, not concrete internals.
